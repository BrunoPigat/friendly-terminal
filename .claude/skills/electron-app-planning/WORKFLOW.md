# 実行ワークフロー

このドキュメントでは、デスクトップアプリ（Electron）企画スキルの詳細な実行手順を説明します。

## 実行手順

### 1. 初期ヒアリング

ユーザーから以下の情報を収集：

#### 必須項目

- **アプリのテーマ・分野**: どのような領域か（生産性、クリエイティブ、開発ツール、ユーティリティなど）
- **解決したい課題**: ユーザーのどんな問題を解決するか
- **対象プラットフォーム**: Windows / macOS / Linux（複数選択可）
- **配布方法**: 自社サイト / Microsoft Store / Mac App Store / 社内配布

#### オプション項目

- **ターゲットユーザー**: 想定している利用者層
- **収益モデル**: 買い切り / サブスクリプション / フリーミアム / 無料
- **ネイティブ機能要件**: ファイルシステム、システムトレイ、グローバルショートカットなど
- **参考アプリ**: 参考にしているデスクトップアプリ
- **オフライン対応**: インターネット接続なしでの動作要件

### 2. アイデアブレインストーミング

#### 2-1. 利用シーンの深掘り

- ペルソナ設定（具体的なユーザー像）
- デスクトップでの作業フロー分析
- Webアプリではなくデスクトップアプリが適している理由

#### 2-2. アイデアの発散

- 3〜5パターンのアプリコンセプトを提案
- 各アイデアの独自性
- Webアプリ/モバイルアプリとの差別化

### 3. 市場・競合分析

#### 3-1. 競合アプリの調査

- WebSearchで類似デスクトップアプリを調査
- 競合の機能・価格・評判を分析
- 差別化ポイントの特定

#### 3-2. プラットフォーム別市場分析

- Windows市場の規模と特性
- macOS市場の規模と特性
- Linux市場の規模と特性（必要に応じて）

### 4. MVP機能設計

#### 4-1. コア機能の定義

- 必須機能（Must Have）: 3〜5個に絞る
- 重要機能（Should Have）
- あったらいい機能（Nice to Have）

#### 4-2. UI/UX設計

- メインウィンドウ設計（サイズ、リサイズ可否）
- メニューバー/コンテキストメニュー
- システムトレイ統合
- キーボードショートカット
- OS固有UIガイドライン対応

#### 4-3. データ設計

- ローカルデータ保存（SQLite、JSON、electron-store等）
- クラウド同期（必要に応じて）
- 設定ファイルの保存場所

### 5. アーキテクチャ・セキュリティ設計

#### 5-1. プロセス設計

| プロセス | 責務 |
|---------|------|
| Main Process | ウィンドウ管理、ファイル操作、システム連携 |
| Renderer Process | UI描画、ユーザー操作処理 |
| Preload Script | 安全なAPI公開（contextBridge） |

#### 5-2. セキュリティ設計

```javascript
// 必須のセキュリティ設定
{
  webPreferences: {
    nodeIntegration: false,      // 必須
    contextIsolation: true,      // 必須
    sandbox: true,               // 推奨
    preload: path.join(__dirname, 'preload.js')
  }
}
```

#### 5-3. IPC（プロセス間通信）設計

- メイン⇔レンダラー間の通信設計
- preloadスクリプトでの安全なAPI公開
- 入力値の検証

#### 5-4. ネイティブ機能設計

| 機能 | 使用API | 備考 |
|------|---------|------|
| ファイル操作 | fs (via preload) | ダイアログ経由推奨 |
| システムトレイ | Tray | 常駐アプリ向け |
| 通知 | Notification | OS通知連携 |
| グローバルショートカット | globalShortcut | 要検討 |
| 自動起動 | app.setLoginItemSettings | 設定で選択可能に |

### 6. 配布・アップデート戦略

#### 6-1. ビルド・パッケージング

- ビルドツール: electron-builder / electron-forge
- コード署名（Windows/macOS）
- プラットフォーム別インストーラー

| OS | インストーラー形式 |
|----|--------------------|
| Windows | NSIS (.exe) / MSI / AppX |
| macOS | DMG / PKG / Mac App Store |
| Linux | AppImage / deb / rpm / snap |

#### 6-2. 自動アップデート

- electron-updater設定
- 更新サーバーの選定（GitHub Releases、S3など）
- アップデート通知UI

#### 6-3. 配布チャネル

- 自社Webサイト
- GitHub Releases
- Microsoft Store申請
- Mac App Store申請（Sandboxing要件）

### 7. 実装ロードマップ作成

#### 7-1. Phase 1: MVP開発（1〜2ヶ月）

- コア機能の実装
- 基本的なUI
- 単一プラットフォームでの動作確認

#### 7-2. Phase 2: クロスプラットフォーム対応（2〜4週間）

- 全プラットフォームでのテスト
- OS固有の調整
- インストーラー作成
- コード署名

#### 7-3. Phase 3: 配布・改善（公開後）

- 配布チャネルへの公開
- 自動アップデート機能
- ユーザーフィードバック対応

### 8. 企画書の作成と保存

全ての分析・設計内容を企画書としてまとめる：

1. 適切なファイル名を生成（例: `electron-app-plans/markdown-editor.md`）
2. 企画書を`electron-app-plans/`ディレクトリに保存
3. ファイルパスをユーザーに通知

### 9. 次のステップの提案

企画完了後、以下のアクションを提案：

- **プロトタイピング**: 基本的なウィンドウの作成
- **技術検証**: ネイティブ機能の動作確認
- **デザイン**: UIモックアップ作成（Figma等）
- **開発着手**: コーディング開始
- **署名準備**: コード署名証明書の取得

## 実行例

### ユーザー入力例

```
Markdownエディタを作りたいです。
リアルタイムプレビューができて、ローカルファイルを編集できるものです。
Windows と macOS で動かしたいです。
```

### 実行フロー

1. 初期ヒアリング（追加質問：配布方法、収益モデル）
2. アイデアブレインストーミング（3パターン提案）
3. ユーザーが選択したアイデアで競合調査（WebSearch使用）
4. MVP機能設計（エディタ、プレビュー、ファイル保存の3機能）
5. アーキテクチャ設計（IPC、セキュリティ）
6. 開発ロードマップ作成（6週間計画）
7. 企画書を`electron-app-plans/markdown-editor.md`に保存
8. 次のステップを提案（プロトタイプ作成、UIデザイン）
